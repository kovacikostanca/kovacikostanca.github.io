SaaS Revenue Analysis

A full intermediate SQL project — churn detection, MRR calculation, ARPU by segment, and at-risk customer identification. Built on a subscription database designed from scratch in PostgreSQL.

The queries that catch what standard reporting misses.

What is this Project?

You're a data analyst at a SaaS company. The CEO wants to understand why revenue is growing but profit is shrinking. You'll use SQL to find out — by calculating MRR, identifying churned customers, flagging at-risk accounts, and breaking down revenue by customer segment.

Everything is built from scratch — schema design, table creation, realistic data insertion, and 5 business queries using SQL concepts that go beyond basic SELECT statements.

Phase 1 - Database Schema

The 4 tables and why they're structured this way

customers — who they are and which segment they belong to (Startup, SMB, Enterprise). The segment column is what drives the CASE WHEN and GROUP BY queries later.

plans — the 3 pricing tiers: Starter ($29/mo), Pro ($79/mo), Enterprise ($199/mo). A reference table — rarely changes, but referenced by every subscription.

subscriptions — the most important table. Links customers to plans, tracks start and end dates, and holds the status (active, churned, paused). One customer can have multiple subscription rows over time — this is how upgrades and churn are tracked.

payments — every payment event: paid, failed, or refunded. Links to subscriptions, not directly to customers. Finding the gap between subscriptions and payments is exactly where LEFT JOIN becomes essential.

Key schema concept — nullable end_date
The end_date column in subscriptions has no NOT NULL constraint (intentionally). An active subscription has no end date yet, so it stores NULL. When a customer churns, we fill it in. This NULL becomes part of the business logic: WHERE end_date IS NULL finds active customers, WHERE end_date IS NOT NULL finds churned ones.

The data stories — 10 customers, 4 profiles

Healthy (Alice, David, Grace, James)
Active subscriptions, consistent paid payments every month. The baseline.
Churned (Bob, Carol)
Cancelled subscriptions with end_date filled in. Bob left in August, Carol in September.
At-risk (Emma, Henry)
Active subscriptions but most recent payment failed. The query that catches these is the most valuable one in the project.
Upgraders (Frank, Iris)
Started on Starter plan, moved to Pro. Each has two subscription rows — one churned, one active. Real upgrade tracking.

Phase 2 — Creating the Tables

4 tables, created in dependency order. The new concepts introduced here are CHECK constraints with IN(), nullable columns, and multiple foreign keys on a single table.

Table 1 — customers

01_customers.sql
No dependencies
CREATE TABLE customers (
    customer_id   SERIAL        PRIMARY KEY,
    name          VARCHAR(100)  NOT NULL,
    email         VARCHAR(150)  UNIQUE NOT NULL,
    company       VARCHAR(100)  NOT NULL,
    segment       VARCHAR(20)   NOT NULL
                  CHECK (segment IN ('Startup', 'SMB', 'Enterprise')),
    signup_date   DATE          NOT NULL
);
segment CHECK (IN (...))
CHECK with IN restricts this column to only 3 valid values. If you try to insert 'Mid-Market' or a typo, PostgreSQL rejects it instantly — data quality enforced at the schema level, not the application level.
signup_date DATE
When the customer first registered — different from when their subscription started. Tracking both reveals your sales cycle length. A customer might sign up today but not pay for a week.

Table 2 — plans

02_plans.sql
No dependencies
CREATE TABLE plans (
    plan_id         SERIAL        PRIMARY KEY,
    plan_name       VARCHAR(50)   NOT NULL UNIQUE,
    billing_cycle   VARCHAR(10)   NOT NULL
                    CHECK (billing_cycle IN ('monthly', 'annual')),
    monthly_price   NUMERIC(8,2)  NOT NULL CHECK (monthly_price > 0)
);
monthly_price CHECK (> 0)
Two constraints on one column — NOT NULL and a value check. Even annual plans store their monthly equivalent here. This is how MRR (Monthly Recurring Revenue) stays comparable across billing cycles.

Table 3 — subscriptions

03_subscriptions.sql
Needs customers + plans
CREATE TABLE subscriptions (
    subscription_id  SERIAL       PRIMARY KEY,
    customer_id      INT          NOT NULL,
    plan_id          INT          NOT NULL,
    start_date       DATE         NOT NULL,
    end_date         DATE,        NULL,        -- NULL means still active
    status           VARCHAR(10)  NOT NULL
                     CHECK (status IN ('active', 'churned', 'paused')),

    CONSTRAINT fk_customer
        FOREIGN KEY (customer_id) REFERENCES customers(customer_id),

    CONSTRAINT fk_plan
        FOREIGN KEY (plan_id) REFERENCES plans(plan_id)
);
end_date DATE
(no NOT NULL)
Intentionally nullable. Active subscriptions have no end date yet — NULL here is meaningful business data, not missing data. Churned subscriptions get this filled in. WHERE end_date IS NULL finds all active customers.
Two FOREIGN KEYs
This table bridges customers and plans. Both parent tables must exist before this one can be created. This is why subscriptions is always created third in the dependency order.

Table 4 — payments

04_payments.sql
Needs subscriptions
CREATE TABLE payments (
    payment_id       SERIAL        PRIMARY KEY,
    subscription_id  INT           NOT NULL,
    payment_date     DATE          NOT NULL,
    amount           NUMERIC(8,2)  NOT NULL CHECK (amount > 0),
    payment_status   VARCHAR(10)   NOT NULL
                     CHECK (payment_status IN
                     ('paid', 'failed', 'refunded')),

    CONSTRAINT fk_subscription
        FOREIGN KEY (subscription_id)
        REFERENCES subscriptions(subscription_id)
);
FK to subscriptions (not customers)
Payments link to subscriptions, not directly to customers. To trace a payment back to a customer you JOIN through subscriptions. This is a 3-table chain — and it's exactly the pattern the business queries use.
payment_status CHECK (IN (...))
Three states: paid (money received), failed (card declined), refunded (money returned). Failed payments still get a row — you need a record that an attempt was made. The status tells you the outcome.

Phase 4 — Business Queries

5 business questions, 5 queries. Each one introduces a SQL concept that goes beyond the beginner project — and each one answers a question a real SaaS analyst would be asked on their first week of the job.

Query 1 — MRR by plan

Business question: "How much Monthly Recurring Revenue does each plan generate? Are any plans underperforming?"
New concept — HAVING
WHERE filters rows before grouping. HAVING filters groups after grouping. You cannot use WHERE on a SUM or COUNT result — those values don't exist yet when WHERE runs. HAVING runs after GROUP BY, so the aggregates are already calculated.
query_01_mrr_by_plan.sql
HAVING
SELECT
    p.plan_name,
    p.monthly_price,
    COUNT(s.subscription_id)     AS active_subscribers,
    SUM(p.monthly_price)         AS mrr
FROM subscriptions s
JOIN plans p ON s.plan_id = p.plan_id
WHERE s.status = 'active'
GROUP BY p.plan_name, p.monthly_price
HAVING COUNT(s.subscription_id) > 1
ORDER BY mrr DESC;
Expected result
plan_namemonthly_priceactive_subscribersmrr
Enterprise199.002398.00
Pro79.004316.00
Starter29.00258.00
Business insightEnterprise generates the most MRR per customer by far — $199 per subscriber vs $79 for Pro and $29 for Starter. With only 2 Enterprise customers matching the revenue of 4 Pro customers combined, the growth strategy is clear: move customers up-tier, not just acquire more.

Query 2 — Churn analysis

Business question: "Which customers churned? And which active customers have never made a successful payment?"
New concept — LEFT JOIN + IS NULL
A regular JOIN only returns rows where a match exists in both tables — subscriptions with no payments would disappear silently. LEFT JOIN keeps ALL rows from the left table and shows NULL where no match exists. Filtering for IS NULL then finds exactly the rows with no counterpart — the ghost subscribers.
query_02_churn_analysis.sql
LEFT JOIN
-- Formally churned customers
SELECT
    c.name, c.company, c.segment,
    s.end_date   AS churn_date,
    p.plan_name  AS last_plan
FROM subscriptions s
JOIN customers c  ON s.customer_id = c.customer_id
JOIN plans p      ON s.plan_id     = p.plan_id
WHERE s.status = 'churned'
  AND s.end_date IS NOT NULL
ORDER BY s.end_date DESC;

-- Active subs with zero successful payments (LEFT JOIN)
SELECT
    c.name, c.company, s.start_date, p.plan_name
FROM subscriptions s
JOIN customers c   ON s.customer_id    = c.customer_id
JOIN plans p       ON s.plan_id        = p.plan_id
LEFT JOIN payments pay
    ON  s.subscription_id = pay.subscription_id
    AND pay.payment_status = 'paid'
WHERE s.status = 'active'
  AND pay.payment_id IS NULL;

Query 3 — ARPU by segment

Business question: "What is the average revenue per user by customer segment? Which segment is most valuable?"
New concept — CASE WHEN
Think of CASE WHEN as an IF statement inside SQL. PostgreSQL evaluates each WHEN condition top to bottom and returns the first THEN value that matches. ELSE is the fallback. You can use aggregate functions inside CASE WHEN — it runs after GROUP BY so the aggregates are already calculated per group.
query_03_arpu_by_segment.sql
CASE WHEN
SELECT
    c.segment,
    COUNT(DISTINCT c.customer_id)   AS total_customers,
    ROUND(AVG(p.monthly_price), 2) AS arpu,
    SUM(p.monthly_price)            AS segment_mrr,
    CASE
        WHEN AVG(p.monthly_price) >= 150 THEN 'High value'
        WHEN AVG(p.monthly_price) >= 60  THEN 'Mid value'
        ELSE                                   'Low value'
    END                              AS value_tier
FROM subscriptions s
JOIN customers c ON s.customer_id = c.customer_id
JOIN plans p     ON s.plan_id     = p.plan_id
WHERE s.status = 'active'
GROUP BY c.segment
ORDER BY arpu DESC;

Business insightEnterprise customers are worth 6.8× more per user than SMB. The value_tier column was created entirely from data — it didn't exist in any table. This is how analysts add business logic directly in SQL without touching the database schema.

Query 4 — At-risk customers

Business question: Which customers are at risk — active but haven't paid recently?.
New concept — Subquery
A query inside another query. The inner query runs first and produces a result. The outer query uses that result like a regular table. JOIN (...) AS alias creates a temporary derived table on the fly — it doesn't exist in your database, it's computed during the query execution.
query_04_at_risk_customers.sql
Subquery
SELECT
    c.name, c.company, c.segment,
    p.plan_name,
    last_pay.last_payment_date,
    last_pay.last_status
FROM subscriptions s
JOIN customers c  ON s.customer_id = c.customer_id
JOIN plans p      ON s.plan_id     = p.plan_id
JOIN (
    -- Subquery: most recent payment per subscription
    SELECT
        subscription_id,
        MAX(payment_date) AS last_payment_date,
        payment_status   AS last_status
    FROM payments
    WHERE payment_date = (
        SELECT MAX(payment_date)
        FROM payments p2
        WHERE p2.subscription_id = payments.subscription_id
    )
    GROUP BY subscription_id, payment_status
) AS last_pay ON s.subscription_id = last_pay.subscription_id
WHERE s.status = 'active'
  AND last_pay.last_status = 'failed';

Query 5 — Revenue breakdown by segment

Business question: "What is total revenue collected by segment — broken down by paid, failed, and refunded?"
New concept — Conditional aggregation (CASE WHEN inside SUM)
Instead of filtering rows with WHERE, you include all rows but only sum the ones matching a condition. Rows that don't match contribute 0 instead of being excluded. Three SUM(CASE WHEN) columns scan the same rows but pick up different amounts — equivalent to a pivot table, in a single query.
query_05_revenue_by_segment.sql
Conditional aggregation
SELECT
    c.segment,
    COUNT(DISTINCT c.customer_id)            AS customers,
    SUM(pay.amount)                            AS gross_revenue,
    SUM(CASE WHEN pay.payment_status = 'paid'
             THEN pay.amount ELSE 0 END)    AS collected_revenue,
    SUM(CASE WHEN pay.payment_status = 'failed'
             THEN pay.amount ELSE 0 END)    AS failed_revenue,
    SUM(CASE WHEN pay.payment_status = 'refunded'
             THEN pay.amount ELSE 0 END)    AS refunded_revenue
FROM payments pay
JOIN subscriptions s ON pay.subscription_id = s.subscription_id
JOIN customers c     ON s.customer_id      = c.customer_id
GROUP BY c.segment
ORDER BY collected_revenue DESC;

Key Insights Summary

  • Enterprise ARPU sits at $199 — nearly 4× SMB's $54. Every sales dollar should follow ARPU, not headcount.
  • Carol and Bob churned, but Carol is the only real priority. She was on Pro in the Enterprise segment — $79/month lost and a high chance of winning back. Bob was on Starter, lower urgency.
  • Emma and Henry are your hidden risk. Both show as active but their last payment failed. Standard reporting misses them entirely — they only surface through the subquery.
  • SMB has a 31% payment failure rate — $108 failed against $245 collected. That's not a data quirk, it's a revenue leak that needs a dunning system before it gets worse.
  • Startup is your cleanest segment. $1,022 collected, zero failures, zero refunds. Lower ARPU than Enterprise but zero collection risk — a reliable revenue base.
