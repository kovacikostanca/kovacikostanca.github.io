SQL Table Structure

What is a table, really?

A database is like a folder on your computer.
A table is like an Excel sheet inside that folder.
A row is one record — one customer, one product, one order.
A column is one piece of info — like "name" or "city" or "price".

We created 4 tables, each one storing a different type of information about our store.

What does CREATE TABLE actually do?

It's like drawing an empty grid in Excel and labeling the columns. You're not adding any data yet — you're just telling PostgreSQL:

"Hey, I want a table called customers. It has these columns. Each column holds this type of data."

The order we created the tables — and why it matters

1
customers
No dependencies — goes first
2
products
No dependencies — goes second
3
orders
Needs customers to exist first
4
order_items
Needs orders + products first

Why does the order matter?

Some tables reference other tables (like orders referencing customers). If you try to create orders before customers exists, PostgreSQL will throw an error — it can't reference something that doesn't exist yet.

Phase 2 — Creating the Tables

We created 4 tables in PostgreSQL, one at a time, in dependency order. Below is the SQL for each table with a line-by-line explanation.

Table 1 — customers

01_customers.sql
No dependencies
-- Create customers table
CREATE TABLE customers (
    customer_id  SERIAL        PRIMARY KEY,
    name         VARCHAR(100)  NOT NULL,
    email        VARCHAR(150)  UNIQUE NOT NULL,
    region       VARCHAR(50)   NOT NULL,
    city         VARCHAR(100)  NOT NULL
);
What does each line mean?
CREATE TABLE customers
"Make me a new empty table and call it customers." Simple as that.
customer_id SERIAL PRIMARY KEY
SERIAL = a number that counts up automatically (1, 2, 3…). You never type this — PostgreSQL assigns it. PRIMARY KEY = this is the unique ID for each row. No two customers share the same ID.
name VARCHAR(100)
VARCHAR(100) means "text, up to 100 characters". Like a text box with a character limit. Names don't need more than 100 characters.
email VARCHAR(150) UNIQUE NOT NULL
UNIQUE = two customers can't have the same email. NOT NULL = this field is required — you can't leave it blank. Both rules are enforced automatically by PostgreSQL.
( ) and the semicolon ;
The parentheses wrap all column definitions. The semicolon tells PostgreSQL "this statement is finished." Every SQL statement ends with a semicolon.

Table 2 — products

02_products.sql
No dependencies
-- Create products table
CREATE TABLE products (
    product_id    SERIAL        PRIMARY KEY,
    product_name  VARCHAR(200)  NOT NULL,
    category      VARCHAR(50)   NOT NULL,
    sub_category  VARCHAR(50)   NOT NULL
);
What does each line mean?
product_id SERIAL PRIMARY KEY
Same pattern as customers. Every product gets a unique auto-assigned number. Instead of typing "Staples Easy Button" in every order, we just say product_id = 7. Much cleaner.
product_name VARCHAR(200)
200 characters because product names can be long — like "Avery Heavy-Duty View Binder with 2-Inch Slant Ring". We gave it more room than the name column.
category / sub_category
3 categories: Furniture, Technology, Office Supplies. sub_category is more specific — e.g. category = "Technology", sub_category = "Phones". This lets us later compare sales across product types.

Table 3 — orders

03_orders.sql
Needs customers first
-- Create orders table
CREATE TABLE orders (
    order_id     SERIAL        PRIMARY KEY,
    customer_id  INT           NOT NULL,
    order_date   DATE          NOT NULL,
    ship_mode    VARCHAR(50)   NOT NULL,
    discount     NUMERIC(4,2)  DEFAULT 0,

    CONSTRAINT fk_customer
        FOREIGN KEY (customer_id)
        REFERENCES customers(customer_id)
);
What does each line mean?
customer_id INT NOT NULL
INT = a whole number. This holds the ID of the customer who placed this order. Instead of writing "Maria" in every order, we just write her ID number — for example, 3.
order_date DATE
DATE stores just a calendar date — like 2023-06-15. PostgreSQL knows it's a date, so you can later filter by month, year, or date range easily.
discount NUMERIC(4,2) DEFAULT 0
NUMERIC(4,2) = a decimal number, 4 digits total, 2 after the dot. Example: 0.20 = 20% discount. DEFAULT 0 means if you don't mention discount, it automatically becomes 0.
CONSTRAINT fk_customer FOREIGN KEY…
A FOREIGN KEY rule: "the customer_id in this table must exist in the customers table." PostgreSQL will reject any order for a customer that doesn't exist. It's a safety net built into the database.

Table 4 — order_items

04_order_items.sql
Needs orders + products
-- Create order_items table (the bridge table)
CREATE TABLE order_items (
    item_id      SERIAL          PRIMARY KEY,
    order_id     INT             NOT NULL,
    product_id   INT             NOT NULL,
    quantity     INT             NOT NULL CHECK (quantity > 0),
    unit_price   NUMERIC(10,2)   NOT NULL,
    total_sales  NUMERIC(10,2)   NOT NULL,

    CONSTRAINT fk_order
        FOREIGN KEY (order_id)
        REFERENCES orders(order_id),

    CONSTRAINT fk_product
        FOREIGN KEY (product_id)
        REFERENCES products(product_id)
);
What does each line mean?
quantity INT NOT NULL CHECK (quantity > 0)
INT = whole numbers only — you can't sell 2.5 chairs. CHECK (quantity > 0) rejects any row where quantity is 0 or negative. A safety rule built right into the table definition.
unit_price NUMERIC(10,2)
Up to 10 digits with 2 decimal places — enough for any realistic price like 1299.99. Always use NUMERIC (not FLOAT) for money — FLOAT has rounding errors that corrupt financial data over time.
total_sales NUMERIC(10,2)
Pre-calculated: quantity × unit_price × (1 − discount). Stored so our SUM queries later are fast and simple — no math at query time.
Two FOREIGN KEYs
This table links orders to products — that's why it's called a bridge table and why it's always created last. Both order_id and product_id must exist in their parent tables before a row can be inserted here.

Phase 3 — Inserting the Data

CREATE TABLE built the empty grids. INSERT INTO is where we fill them with real data. Same dependency order applies — customers and products first, then orders, then order_items.

Key rule

We never insert the ID column (customer_id, product_id, etc). PostgreSQL assigns it automatically via SERIAL — starting at 1 and counting up. Alice gets customer_id = 1, Bob gets 2, and so on. Remember these numbers — orders will reference them.

Phase 4 — Business Queries

With the database fully populated, we wrote 5 queries — each answering a real business question a retail manager would actually ask.

Business question: "Which region is generating the most revenue? Where should we focus our sales team?"
query_01_revenue_by_region.sql
2-table JOIN
SELECT
    c.region,
    SUM(oi.total_sales)              AS total_revenue,
    COUNT(DISTINCT o.order_id)       AS total_orders,
    ROUND(AVG(oi.total_sales), 2)    AS avg_order_value
FROM order_items oi
JOIN orders o        ON oi.order_id   = o.order_id
JOIN customers c     ON o.customer_id = c.customer_id
GROUP BY c.region
ORDER BY total_revenue DESC;
Business insightThe East leads in revenue despite matching West in order count — meaning East customers spend more per order. That's a story worth telling to a manager.
Business question: "Which products bring in the most money? What should we keep in stock no matter what?"
query_02_top_products.sql
JOIN + LIMIT
SELECT
    p.product_name,
    p.category,
    SUM(oi.total_sales)   AS total_revenue,
    SUM(oi.quantity)      AS units_sold
FROM order_items oi
JOIN products p   ON oi.product_id = p.product_id
GROUP BY p.product_name, p.category
ORDER BY total_revenue DESC
LIMIT 5;
Business insightThe iPhone dominates revenue despite selling only 2 units — high price per item. The chair sold 3 units but ranks 4th. Volume alone doesn't equal revenue. Price matters.
Business question: "Which product category brings in the most revenue and sells the most units?"
query_03_category_breakdown.sql
Division in SQL
SELECT
    p.category,
    SUM(oi.total_sales)               AS total_revenue,
    SUM(oi.quantity)                  AS units_sold,
    COUNT(DISTINCT oi.order_id)       AS num_orders,
    ROUND(
        SUM(oi.total_sales) /
        COUNT(DISTINCT oi.order_id)
    , 2)                               AS revenue_per_order
FROM order_items oi
JOIN products p   ON oi.product_id = p.product_id
GROUP BY p.category
ORDER BY total_revenue DESC;
Business insightOffice Supplies sells 15 units — more than the other two categories combined — but generates the least revenue. Technology sells only 6 units but earns the most. These categories need completely different strategies.
Business question: "Who are our most valuable customers? Who should we reach out to first for loyalty offers?"
query_04_best_customers.sql
3-table JOIN
SELECT
    c.name,
    c.region,
    c.city,
    COUNT(DISTINCT o.order_id)   AS total_orders,
    SUM(oi.total_sales)           AS lifetime_value
FROM customers c
JOIN orders o        ON c.customer_id = o.customer_id
JOIN order_items oi  ON o.order_id    = oi.order_id
GROUP BY c.name, c.region, c.city
ORDER BY lifetime_value DESC;
Business insightJames from Philadelphia spent $1,304 in a single order — your highest value customer. In a real business this person gets a personal email, a loyalty reward, or early access. Data drives that decision.
Business question: "How did our revenue grow month by month in 2023? Are there slow months we should prepare for?"
query_05_monthly_trend.sql
DATE functions
SELECT
    TO_CHAR(o.order_date, 'YYYY-MM')   AS month,
    COUNT(DISTINCT o.order_id)         AS orders_placed,
    SUM(oi.total_sales)                AS monthly_revenue
FROM orders o
JOIN order_items oi  ON o.order_id = oi.order_id
GROUP BY TO_CHAR(o.order_date, 'YYYY-MM')
ORDER BY month ASC;
Business insightNovember is the highest revenue month — Black Friday effect, even in a small dataset. April is the weakest. In a real business you'd use this to plan inventory, promotions, and staffing months in advance.

Key Insights Summary

  • East region leads in revenue despite matching West in order count — higher spend per order, not more orders.
  • Technology dominates revenue with the fewest units sold — price per unit matters more than quantity. Critical inventory insight.
  • Office Supplies sells the most units but generates the least revenue. Needs a different pricing or bundling strategy.
  • James Thomas is the highest lifetime value customer — $1,304 in one order. Loyalty programs should target profiles like his.
  • November is peak month, April is the danger month — seasonal patterns that should drive planning for the following year.
